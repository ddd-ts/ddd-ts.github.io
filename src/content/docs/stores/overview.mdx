---
title: Store contract
description: The Store and TransactionPerformer contracts, and how DDD-TS talks to any backing database.
---

DDD-TS keeps its domain code free of database details by funneling every
persistence operation through two small interfaces, exposed from
`@ddd-ts/core`:

- `Store<T>`: a transactional key/value collection of a single aggregate
  type.
- `TransactionPerformer`: runs a callback inside a transaction.

Every infrastructure package ships an implementation of these two
interfaces. The in-memory adapter is the reference one; Firestore (and
future adapters) must pass the same test suite.

## `Store<T>`

```ts
interface Store<T> {
  load(id: string, trx?: Transaction): Promise<T | undefined>;
  loadAll(ids: string[], trx?: Transaction): Promise<T[]>;
  save(model: T, trx?: Transaction): Promise<void>;
  delete(id: string, trx?: Transaction): Promise<void>;
}
```

Stores are aggregate-specific: one store per type. A store is *not* an event
store; it persists a snapshot of a value (typically a projection's read
model). Event sourcing has its own contracts, layered on top.

Passing a `Transaction` opts that call into a larger transaction. Without
one, the operation runs in its own transaction.

## `TransactionPerformer`

```ts
interface TransactionPerformer {
  perform<T>(action: (trx: Transaction) => Promise<T>): Promise<T>;
}
```

A `perform` call gives you a transaction object. Any store call passed that
object joins the same transaction; you can mix and match across stores from
the same provider.

```ts
import { InMemoryTransactionPerformer } from "@ddd-ts/store-inmemory";

const transactions = new InMemoryTransactionPerformer();

await transactions.perform(async (trx) => {
  const a = await accounts.load(fromId, trx);
  const b = await accounts.load(toId, trx);
  a.balance -= 50;
  b.balance += 50;
  await accounts.save(a, trx);
  await accounts.save(b, trx);
});
```

Implementations of `TransactionPerformer` also expose a serializer-aware
`TransactionEffect` API for "do this only if the transaction commits" hooks,
useful for publishing domain events to an outbound bus.

## ConcurrencyError

When two writers race for the same key, the loser throws
`ConcurrencyError` (also re-exported from `@ddd-ts/core`). It carries the
identifier and the expected vs. actual revision so callers can decide whether
to retry, surface the error, or fold it into a conflict resolution.

## Why two interfaces and not one ORM?

This is the smallest surface that lets the domain code stay independent of the
backing store. Anything richer (change streams, joins, full-text search)
belongs in a *projection*, not in the store contract. The store is for
"load this, save that", nothing more.
