---
title: "@ddd-ts/freeze"
description: Pin the serialized shape of a class so future refactors can't silently change persisted data.
---

`@ddd-ts/freeze` lets you turn the type of any value into a compile-time
literal that is then committed to your repository. The next time you build the
project, TypeScript checks that the *current* type still matches the frozen
one — and fails the build if it doesn't.

It is the safety net for serialized shapes: event payloads, snapshot states,
projection read models. Anything that lives on disk.

## Why this exists

Schemas drift. A `string` becomes `string | null`. An enum gains a variant. A
nested class is renamed. Each of those changes is invisible in code review,
but they all break the production data layer the next time you deploy.

`freeze(...)` makes the change visible by failing the type checker:

```
error TS2322: Type '{ amount: number; }' is not assignable to type '{ amount: number; currency: string; }'.
```

You then either update the frozen snapshot intentionally (acknowledging the
change is safe / migrated) or change the code back.

## The `freeze` function

```ts
import { freeze } from "@ddd-ts/freeze";
import type { Serialized } from "@ddd-ts/core";

freeze<{
  type: Serialized<DepositedV1>;
  name: "DepositedV1";
}>();
```

The first time you run the CLI, freeze generates a snapshot of the resolved
type and writes it back into the call site as the second type argument:

```ts
freeze<{
  type: Serialized<DepositedV1>;
  name: "DepositedV1";
}, {
  id: string;
  accountId: string;
  amount: number;
  version: 1;
}>();
```

From that point on, any change to `Serialized<DepositedV1>` that doesn't match
the frozen literal becomes a compile error.

## The `@Freeze` decorator

For shape and event classes, the decorator form is more compact — it freezes
the serialized form of every instance method that returns it:

```ts
import { Freeze } from "@ddd-ts/freeze";

@Freeze({ name: "Deposited" })
class Deposited extends EsEvent("Deposited", { ... }) {}
```

## The CLI

`@ddd-ts/freeze` ships a `freeze` binary that scans a TypeScript project,
finds every unfrozen `freeze<...>()` call, and writes back the resolved type.

```sh
pnpm exec freeze --project ./tsconfig.json
```

Run it locally when you change a frozen shape on purpose. Run it in CI
without the `--write` flag to verify everything is still frozen:

```sh
pnpm exec freeze --project ./tsconfig.json --check
```

A CI failure means somebody changed a persisted shape without updating its
snapshot — which is exactly when you want to look at the diff.

## When to freeze

- **Always** freeze event serializers — `Serialized<DepositedV1>`, …
- Freeze snapshot serializers if you snapshot aggregates.
- Freeze projection read models whose shape lives on disk.

You usually don't need to freeze internal value objects unless they are
embedded in something persisted.
